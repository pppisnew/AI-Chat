# AI 聊天助手产品需求文档 (PRD)

**文档版本**：V1.0
**项目名称**：Ai-Chat（本地智能对话助手）

---

## 1. 产品概述和目标用户

### 1.1 产品概述
这是一款基于 Web 技术构建的 AI 对话助手应用。核心定位为“轻量级、隐私优先、体验流畅”。它通过调用大语言模型（LLM）API 提供智能对话服务，采用本地浏览器存储技术实现数据持久化，模拟微信的交互界面风格，降低用户学习成本，提供亲切的使用体验。

### 1.2 目标用户
*   **普通用户**：希望体验 AI 对话乐趣，注重隐私，不希望聊天记录上传至云端服务器的用户。
*   **轻度开发者**：需要一个简洁的 UI 来测试 Prompt 效果，或作为个人知识库前端的用户。
*   **特定场景使用者**：需要模拟即时通讯软件交互逻辑进行特定任务（如客服话术演练）的人群。

## 2. 详细的功能列表

### 2.1 核心对话功能
*   **文本输入与发送**：支持文本框输入，按 Enter 键或点击发送按钮提交。
*   **流式输出 (SSE)**：AI 回答以打字机效果逐字显示，而非等待全部生成完毕一次性展示，提升交互感知速度。
*   **多轮对话记忆**：系统自动维护上下文窗口，将历史对话记录附带在每次请求中，确保 AI 知道前文内容。
*   **Markdown 渲染**：支持渲染代码块、列表、加粗等富文本格式。

### 2.2 会话管理功能
*   **新建聊天**：一键开启新的对话会话，重置上下文，左侧会话列表新增一个空白会话。
*   **清空记录**：
    *   **清空当前会话**：删除当前聊天窗口的所有消息。
    *   **清空所有数据**：提供彻底清除本地存储的功能（需二次确认）。
*   **历史会话列表**：左侧栏展示历史创建的会话列表，支持切换查看。
*   **会话标题自动生成**：根据用户第一句提问自动生成会话标题（如截取前几个字）。

### 2.3 数据持久化
*   **本地存储**：利用浏览器数据库（LocalStorage）保存所有聊天记录。
*   **断点续聊**：刷新浏览器页面或关闭后重新打开，聊天记录完整保留，可继续对话。

### 2.4 系统设置
*   **API Key 配置**：用户输入自己的 API Key（支持 OpenAI/其他兼容接口），Key 仅保存在本地，不上传服务器。
*   **模型选择**：使用智谱的GLM-4.7-Flash
*   **Prompt 预设**：支持设置系统级 Prompt（如“你是一个翻译官”），该设定在新建聊天时生效。

## 3. 功能优先级（MVP 和后续版本）

### 3.1 MVP (Minimum Viable Product) 版本 - 第一阶段
**目标**：跑通核心流程，实现基本的对话与存储。

| 优先级 | 功能模块 | 功能描述 |
| :--- | :--- | :--- |
| **P0** | 对话交互 | 用户发送文本，AI 流式回复 |
| **P0** | 上下文记忆 | 自动携带最近 N 条历史记录发送给 API |
| **P0** | 本地存储 | 刷新页面后记录不丢失 (LocalStorage 即可) |
| **P0** | 界面布局 | 微信风格的双栏布局（左侧列表+右侧聊天） |
| **P0** | 基础设置 | API Key 输入与本地保存 |
| **P1** | 会话管理 | 新建聊天、切换会话、删除当前会话 |

### 3.2 V1.5 版本 - 第二阶段
**目标**：优化体验，增强管理能力。

| 优先级 | 功能模块 | 功能描述 |
| :--- | :--- | :--- |
| **P1** | 数据管理 | 迁移至 IndexedDB，支持大量数据存储；支持导出聊天记录。 |
| **P2** | 界面优化 | 消息气泡复制按钮、代码块一键复制、滚动到底部按钮。 |
| **P2** | Prompt 管理 | 自定义 System Prompt 并保存为模板。 |
| **P2** | 响应式设计 | 适配移动端浏览器视图。 |

## 4. 界面设计要求

**设计风格**：高度模仿微信 PC 版/网页版，采用清新简洁的色调。

### 4.1 布局结构
*   **左侧栏（会话区）**：
    *   顶部：搜索框（V1.5）+ “新建聊天”按钮。
    *   中部：会话列表，显示头像（AI 默认头像）、标题、最后一条消息摘要、时间。
    *   底部：设置入口（齿轮图标）。
*   **右侧栏（聊天区）**：
    *   顶部：显示当前会话标题，右侧包含“清空当前记录”图标。
    *   中部：消息流区域。
        *   **用户消息**：绿色气泡，靠右对齐。
        *   **AI 消息**：白色/灰色气泡，靠左对齐，带头像。
        *   **加载状态**：显示“正在输入...”动画。
    *   底部：输入区域（文本框 + 发送按钮）。

### 4.2 交互细节
*   **流式效果**：AI 回复时，文本逐字出现，页面自动滚动到底部，直到生成完毕。
*   **发送动作**：点击发送后，输入框清空，用户消息立即上屏，AI 消息区域先显示 Loading 状态。
*   **错误提示**：若 API 报错（如 Key 无效），以系统通知的形式在聊天窗口中央提示。

## 5. 技术栈建议

考虑到需要快速开发、响应式布局及本地存储需求，建议采用以下技术栈：

### 5.1 前端框架
*   **React / Vue 3**：组件化开发，生态丰富，适合构建交互复杂的单页应用。
*   **Tailwind CSS**：原子化 CSS 框架，非常适合快速搭建仿微信的 UI 布局。

### 5.2 数据存储
*   **IndexedDB (推荐)**：浏览器内置数据库，存储容量大（通常 > 50MB），适合保存大量聊天历史。
    *   *库推荐*：`Dexie.js` (对 IndexedDB 的封装，API 友好)。
*   **LocalStorage**：仅用于存储简单的配置信息（如 API Key、主题设置）。

### 5.3 AI 接口对接
*   **Fetch API / Axios**：处理 HTTP 请求。
*   **流式处理**：前端需处理 `ReadableStream`，使用 `reader.read()` 循环读取数据块，实时更新 UI。

### 5.4 Markdown 处理
*   `react-markdown` 或 `marked`：用于解析 Markdown 文本。
*   `highlight.js`：用于代码高亮显示。

## 6. 非功能性需求

### 6.1 性能要求
*   **首屏加载**：首屏渲染时间 < 1秒。
*   **滚动流畅**：当聊天记录超过 1000 条时，列表滚动应保持 60fps，建议使用虚拟列表技术（如 `react-window`）优化长列表渲染。
*   **流式延迟**：接收到 API 数据流后，界面渲染延迟 < 50ms。

### 6.2 安全要求
*   **API Key 隔离**：API Key 必须存储在本地（LocalStorage 或 IndexedDB），严禁在任何情况下上传至第三方服务器（除非用户自建后端代理）。
*   **XSS 防护**：渲染 Markdown 内容时必须进行过滤，防止用户输入恶意脚本导致 XSS 攻击（使用 `DOMPurify` 清洗 HTML）。

### 6.3 可用性与容错
*   **离线提示**：检测到网络断开时，输入框上方显示“网络已断开”提示。
*   **数据备份**：建议在 V1.5 版本提供“导出 JSON”功能，防止用户清除浏览器数据导致丢失。

### 6.4 兼容性
*   支持 Chrome, Edge, Safari, Firefox 最新两个主版本。
*   移动端浏览器自适应适配。
